# MCP-AQL Specification

**Version:** 1.0.0-draft
**Status:** Draft
**Last Updated:** 2026-01-05

## Abstract

MCP-AQL (Model Context Protocol - Advanced Agent API Adapter Query Language) is a protocol specification that consolidates multiple MCP tools into semantic CRUDE endpoints, providing significant token reduction while enabling runtime operation discovery through introspection.

## Status of This Document

This is a draft specification subject to change. Feedback is welcome via the [specification repository](https://github.com/MCPAQL/spec).

## 1. Introduction

### 1.1 Purpose

MCP-AQL addresses the token efficiency challenge in LLM-MCP interactions. Traditional MCP implementations expose discrete tools, each requiring schema definition in the model's context. With 50+ tools, this can consume approximately 30,000 tokens before any work begins.

MCP-AQL consolidates operations into five semantic endpoints, reducing token overhead by approximately 85-96% while maintaining full functionality.

### 1.2 Goals

1. **Token Efficiency** - Minimize context consumed by tool definitions
2. **Semantic Clarity** - Group operations by their effect on system state
3. **Runtime Discovery** - Enable operation discovery via introspection
4. **Flexibility** - Support multiple endpoint modes for different use cases
5. **Extensibility** - Allow domain-specific operation extensions

### 1.3 Scope

This specification defines:
- The CRUDE endpoint pattern
- Request and response formats
- Core operations that implementations MUST support
- The introspection system for operation discovery
- Conformance requirements

This specification does NOT define:
- Domain-specific operations beyond the core set
- Implementation architecture details
- Transport-layer concerns (handled by MCP)

## 2. Terminology

### 2.1 Key Terms

**CRUDE Pattern**
: The five-endpoint pattern: Create, Read, Update, Delete, Execute

**Element**
: A discrete unit of functionality managed by an MCP-AQL server (e.g., persona, skill, agent)

**Operation**
: A specific action that can be performed (e.g., `create_element`, `list_elements`)

**Endpoint**
: One of the five CRUDE categories to which operations are routed

**Introspection**
: The ability to query available operations and types at runtime

### 2.2 Requirement Levels

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in [RFC 2119](https://www.ietf.org/rfc/rfc2119.txt).

## 3. Protocol Overview

### 3.1 Architecture

```
┌─────────────────────────────────────────────────────────────┐
│                        MCP Client                           │
│                     (LLM / Agent)                           │
└─────────────────────────┬───────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────────┐
│                     MCP-AQL Layer                           │
│  ┌──────────────────────────────────────────────────────┐   │
│  │              CRUDE Endpoints                         │   │
│  │   CREATE │ READ │ UPDATE │ DELETE │ EXECUTE          │   │
│  └──────────────────────────────────────────────────────┘   │
│                          │                                  │
│  ┌──────────────────────────────────────────────────────┐   │
│  │              Operation Router                        │   │
│  │   Routes operations to correct handlers              │   │
│  └──────────────────────────────────────────────────────┘   │
│                          │                                  │
│  ┌──────────────────────────────────────────────────────┐   │
│  │              Introspection System                    │   │
│  │   Provides runtime operation discovery               │   │
│  └──────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────────┐
│                    Domain Handlers                          │
│            (Element CRUD, Execution, etc.)                  │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 Request Flow

1. Client sends request to MCP-AQL endpoint
2. Router extracts `operation` field
3. Router validates operation belongs to receiving endpoint
4. Parameters are extracted and validated
5. Handler executes operation
6. Response returned to client

### 3.3 Endpoint Modes

MCP-AQL supports two operational modes:

**CRUDE Mode (5 endpoints)**
```
mcp_aql_create   - CREATE operations
mcp_aql_read     - READ operations
mcp_aql_update   - UPDATE operations
mcp_aql_delete   - DELETE operations
mcp_aql_execute  - EXECUTE operations
```

**Single Mode (1 endpoint)**
```
mcp_aql - All operations through unified entry point
```

Implementations MUST support at least one mode. Implementations SHOULD support both.

## 4. Request Format

### 4.1 Standard Request Structure

```typescript
interface OperationInput {
  operation: string;              // REQUIRED: Operation to perform
  element_type?: ElementType;     // OPTIONAL: Element type context
  params?: Record<string, any>;   // OPTIONAL: Operation parameters
}
```

### 4.2 Parameter Resolution

Parameters MAY appear at multiple levels. Resolution follows this precedence:

1. `params.<parameter_name>`
2. Top-level `<parameter_name>`
3. Mapped alternatives (e.g., `elementType` → `element_type`)

### 4.3 Parameter Naming

All public-facing parameters MUST use snake_case naming:
- `element_name` (not `elementName`)
- `element_type` (not `elementType`)

For backward compatibility, implementations SHOULD accept camelCase alternatives.

### 4.4 Request Examples

**Minimal Request:**
```json
{
  "operation": "list_elements",
  "element_type": "persona"
}
```

**Full Request:**
```json
{
  "operation": "create_element",
  "element_type": "persona",
  "params": {
    "element_name": "CodeReviewer",
    "description": "A thorough code reviewer",
    "instructions": "You are a meticulous code reviewer."
  }
}
```

## 5. Response Format

### 5.1 Discriminated Union

All operations return a discriminated union result:

```typescript
type OperationResult = OperationSuccess | OperationFailure;

interface OperationSuccess {
  success: true;
  data: any;      // Operation-specific payload
}

interface OperationFailure {
  success: false;
  error: string;  // Human-readable error message
}
```

### 5.2 Success Response

```json
{
  "success": true,
  "data": {
    "name": "CodeReviewer",
    "type": "persona",
    "description": "A thorough code reviewer",
    "created": "2024-01-15T10:30:00Z"
  }
}
```

### 5.3 Error Response

```json
{
  "success": false,
  "error": "Missing required parameter 'element_name' for operation 'create_element'"
}
```

## 6. CRUDE Endpoints

### 6.1 Endpoint Summary

| Endpoint | Read-Only | Destructive | Description |
|----------|-----------|-------------|-------------|
| CREATE | false | false | Additive operations |
| READ | true | false | Query operations |
| UPDATE | false | true | Modification operations |
| DELETE | false | true | Removal operations |
| EXECUTE | false | true | Runtime lifecycle operations |

### 6.2 Operation Routing

Each operation MUST be routed to exactly one endpoint. Implementations MUST reject operations sent to incorrect endpoints in CRUDE mode.

See [CRUDE Pattern Specification](../crude-pattern.md) for detailed endpoint definitions.

## 7. Introspection

### 7.1 Purpose

Introspection enables clients to discover available operations without parsing tool schemas, providing:
- On-demand operation discovery
- Parameter documentation
- Type information
- Usage examples

### 7.2 The `introspect` Operation

**Endpoint:** READ

**Parameters:**
| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `query` | string | Yes | Query type: "operations" or "types" |
| `name` | string | No | Specific item name for details |

### 7.3 Query Types

**List Operations:**
```json
{ "operation": "introspect", "params": { "query": "operations" } }
```

**Operation Details:**
```json
{ "operation": "introspect", "params": { "query": "operations", "name": "create_element" } }
```

**List Types:**
```json
{ "operation": "introspect", "params": { "query": "types" } }
```

**Type Details:**
```json
{ "operation": "introspect", "params": { "query": "types", "name": "ElementType" } }
```

See [Introspection Specification](../introspection.md) for detailed response structures.

## 8. Core Operations

### 8.1 Required Operations

Conforming implementations MUST support these core operations:

**CREATE Endpoint:**
- `create_element` - Create a new element
- `activate_element` - Activate an element

**READ Endpoint:**
- `list_elements` - List elements with filtering
- `get_element` - Retrieve element by name
- `introspect` - Operation and type discovery
- `get_active_elements` - List active elements

**UPDATE Endpoint:**
- `edit_element` - Modify an element
- `deactivate_element` - Deactivate an element

**DELETE Endpoint:**
- `delete_element` - Remove an element

**EXECUTE Endpoint:**
- `execute_agent` - Start agent execution
- `get_execution_state` - Query execution state
- `complete_execution` - Signal completion

### 8.2 Optional Operations

Implementations MAY support additional operations. All operations MUST be discoverable via introspection.

See [Operations Reference](../operations.md) for complete operation documentation.

## 9. Type System

### 9.1 Core Types

**ElementType** (enum)
```
persona | skill | template | agent | memory | ensemble
```

**CRUDEndpoint** (enum)
```
CREATE | READ | UPDATE | DELETE | EXECUTE
```

**OperationResult** (union)
```
OperationSuccess | OperationFailure
```

### 9.2 Type Discovery

All types MUST be discoverable via the `introspect` operation with `query: "types"`.

## 10. Conformance

### 10.1 Conformance Levels

**Level 1: Basic Conformance**
- MUST implement all required operations
- MUST support at least one endpoint mode
- MUST implement introspection
- MUST use standard request/response formats

**Level 2: Full Conformance**
- MUST meet Level 1 requirements
- MUST support both CRUDE and Single endpoint modes
- MUST pass the conformance test suite
- SHOULD provide all optional operations

### 10.2 Claiming Conformance

Implementations claiming MCP-AQL conformance MUST:
1. State their conformance level
2. Document any deviations or extensions
3. Pass the applicable conformance tests

### 10.3 Extensions

Implementations MAY extend MCP-AQL with additional:
- Operations (MUST be introspectable)
- Element types (MUST work with core operations)
- Response fields (MUST NOT break standard fields)

Extensions MUST be documented and MUST NOT conflict with core specification behavior.

## Appendix A: Token Efficiency

### A.1 Measured Token Costs

| Configuration | Tools | Tokens | Reduction |
|---------------|-------|--------|-----------|
| Discrete Tools | 50+ | ~29,600 | Baseline |
| CRUDE Mode | 5 | ~4,300 | ~85% |
| Single Mode | 1 | ~1,100 | ~96% |

### A.2 Introspection Pattern

```
Total context for 10 operations with introspection:
- Tool registration: ~1,100 tokens
- 10 × introspect calls: ~1,500 tokens
- Total: ~2,600 tokens (vs ~29,600 baseline)
```
